How To Use This Guide

Terminal 1: run Gazebo

Use one launch command from Room 315, Full Floor, or Single Robot.

Terminal 2: send commands

Open a new terminal, source the workspace, then use shuttle, switch, stopper, sensor, or robot commands.

Terminal 3: watch state

Use ros2 topic echo for shuttle state, switch state, stopper state, and sensor feedback.

After edits

Run the matching validation command after changing robot config, device YAML, rail CSV files, or rail network YAML.

Recommended workflow

  1. Build once with Setup -> Full build.
  2. Launch the smallest world that matches your test.
  3. Use robots:=none for rail-only debugging.
  4. Use startup shuttle counts for repeatable tests, or runtime add commands for interactive tests.
  5. Watch state topics while sending commands so you can see requested state versus actual delayed state.
  6. Use validation tools before Gazebo when changing CSV, network YAML, or device YAML.

Troubleshooting

  • If Gazebo does not open, source the workspace and launch Room 315 or the full floor before starting standalone shuttle nodes.
  • If set_pose is not ready, check ros2 service list and the Gazebo world name.
  • Room 315 services should use /world/room_315_only/....
  • Full-floor services should use /world/mfja_3rd_floor/....
  • If an add command is rejected, another shuttle may still occupy that start slot.
  • If a shuttle enters WAITING, it is likely blocked by collision avoidance.
  • If a shuttle enters FALLING, send RESET after checking switch routing.
  • If route logic does not change, send switch commands to the rail topic, not directly to visual switch topics.
  • If you edit CSV geometry, run the validation commands before Gazebo testing.
  • If you edit device YAML, rerun the device validator and relaunch Gazebo.
